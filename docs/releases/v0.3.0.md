# humane-swift v0.3.0

## `TimeFormatter`/`SizeFormatter` gain `at:` and fully positional call styles

```swift
Humane.SizeFormatter().string(225_935) // "226 KB"

let timeFormatter = Humane.TimeFormatter()
timeFormatter.string(at: Date().addingTimeInterval(-180), relativeTo: Date()) // "3 minutes ago"
timeFormatter.string(Date().addingTimeInterval(-180), Date())                 // "3 minutes ago"
```

`TimeFormatter.string(at:relativeTo:)` is an additive alias for
`string(for:relativeTo:)` -- `at` is the one parameter name `humane` (Go) and
`humane-ruby` can actually share (Ruby can't call it `for`, a reserved word);
`for:` remains this package's primary, Foundation-matching spelling.
`TimeFormatter.string(_:_:)` and `SizeFormatter.string(_:)` add fully
positional calling, matching `humane` (Go)'s label-free convention.
`humane-ruby`'s `#string` on both formatters picked up equivalent
positional-or-keyword support in the same session. Each addition is a
one-line forward to the existing implementation, not a separate one -- see
`docs/COMMENTS.md`.

README also gained the `approximate: true` example (`"about 15 hours ago"`)
that `humane`'s and `humane-ruby`'s READMEs already had, and made explicit
the principle guiding all of the above: Foundation is the baseline every
default matches exactly, in all three languages; ActionView's vocabulary
(`includeSeconds`, `approximate`, and now `at`) is a layer on top of that
baseline, opt-in, never a replacement for it.

## Upgrading

No breaking changes. Purely additive -- `for:relativeTo:` and
`fromByteCount:` remain the primary spellings and are unaffected.

Written by inspection in the Cowork sandbox (no Swift toolchain there), then
confirmed for real via `swift test` on woodie's Mac -- 38/38 passing.
