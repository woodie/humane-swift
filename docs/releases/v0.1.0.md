# humane-swift v0.1.0

The Swift sibling to [`humane`](https://github.com/woodie/humane) (Go) and
[`humane-ruby`](https://github.com/woodie/humane-ruby): thin wrappers over
`ByteCountFormatter`/`RelativeDateTimeFormatter`, matching their API shape, plus two
additive options for contexts that can't live-refresh a rendered time.

## `Humane.SizeFormatter`

A one-line wrapper around `ByteCountFormatter.string(fromByteCount:countStyle:.file)`
-- Finder's own byte formatting, with no math to reimplement.

```swift
Humane.SizeFormatter().string(fromByteCount: 225_935) // "226 KB"
```

## `Humane.TimeFormatter`

Wraps `RelativeDateTimeFormatter`. Matches its raw output exactly by default:

```swift
Humane.TimeFormatter().string(for: date, relativeTo: now) // "3 minutes ago"
```

Two options, both off by default:

- **`includeSeconds`**: below a minute, collapses to "less than a minute ago"/"in
  less than a minute" instead of an exact second count. Named after ActionView's
  `include_seconds`, which defaults the same way (a static render is stale the
  instant it's produced, so second-level precision there is misleading).
- **`approximate`**: prefixes "about"/"in about" on buckets of an hour or larger, the
  way ActionView's `distance_of_time_in_words` does past that same boundary -- a
  signal that the bucket itself is a rounded value.

```swift
let formatter = Humane.TimeFormatter(approximate: true)
formatter.string(for: date, relativeTo: now) // "about 15 hours ago"
```

## Adopted by `zouk`

`zouk`'s `ScanEntry` used to hand-roll both of the above directly against
`RelativeDateTimeFormatter`, including a manual "less than a minute" clamp -- at 30
seconds, not this library's 60-second default. Switching `zouk` over to
`Humane.TimeFormatter(approximate: true)` fixed that drift as a side effect: anything
30-59 seconds old now reads "less than a minute ago" instead of showing exact
seconds, matching what `humane`/`humane-ruby` have always done. Confirmed via a real
`make test` run against `zouk` -- 46/46 specs passing.

## Known gap

`ByteCountFormatter`'s real output diverges from `humane`/`humane-ruby`'s hand-rolled
2-significant-digit math in a few cases: zero bytes (`"Zero KB"`, not `"0 B"`),
byte-scale values (`"7 bytes"`, not `"7 B"`), and some GB-scale values keep 2 decimal
places instead of 1 (`"5.24 GB"`, not `"5.2 GB"`). `SizeFormatter` here is a direct
passthrough, so it's authoritative on these; whether to correct `humane`/`humane-ruby`
toward exact parity is still an open question (see `docs/COWORK.md`).

## Scope

Finder's `.file` byte-count style, and a numeric (non-calendar-aware) relative time
style -- that's the whole surface area. `allowedUnits`/alternate `countStyle`s and a
`.named` style (`"yesterday"`, calendar-boundary-aware) aren't implemented.

## Upgrading

New library, nothing to upgrade from.
