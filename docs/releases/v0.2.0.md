# humane-swift v0.2.0

## `TimeFormatter` now matches ActionView's bucket table exactly (through "1 day")

```swift
TimeFormatter(approximate: true).string(for: t.addingTimeInterval(-(44 * 60 + 30)), relativeTo: t)
// "about 1 hour ago"
```

`approximate` used to prepend "about "/"in about " onto whatever
`RelativeDateTimeFormatter` returned. That stopped being enough once `approximate`
needed to match ActionView's specific early cutoffs -- the "about 1 hour" bucket
starts at 44 minutes 30 seconds (not 60:00), and "about 2 hours" starts at 89:30, not
90:00 -- because Foundation's own rounding doesn't jump buckets that early. `string`
now computes `distanceInMinutes` (seconds/60, rounded once) and buckets by hand, the
same shape `humane`/`humane-ruby` use, and no longer calls `RelativeDateTimeFormatter`
at all.

Two output changes, both additive to the API (no new parameters):

- `includeSeconds: false`'s collapse cutoff moved from 60 seconds to 30 seconds,
  matching the first row of ActionView's table. `"45 seconds ago"` is now
  `"1 minute ago"` instead of `"less than a minute ago"`.
- `approximate` narrowed from "about" on any bucket >= 1 hour to exactly the
  hour-scale buckets (1 hour, and 2..24 hours). ActionView's own table has no "about
  1 day", so neither does this anymore -- `approximate: true` on a 30-hour-old
  timestamp is now `"1 day ago"`, not `"about 1 day ago"`.

Scoped through the "1 day" row of ActionView's table -- week/month/year buckets are
out of scope, matching this library's "narrow, not full-featured" intent. See
[humane-ruby issue #1](https://github.com/woodie/humane-ruby/issues/1) for the
source table this ports (identically, in the same session, across all three
languages), and `docs/COMMENTS.md` for the full boundary-by-boundary rationale.

Confirmed via a real `swift test` on woodie's Mac -- 35/35 passing.

## Upgrading

No API changes. Two output changes to check for if you depend on exact wording
below 90 seconds or on `approximate`'s day-scale output -- see above. `zouk`'s
`ScanEntry.timeAgo` (the only known `approximate: true` consumer) is unaffected: its
documented usage is hour-scale (`"about 14 hours ago"`), not day-scale.
